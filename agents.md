# Agent Directives for Startup Lab

Startup Lab is a **startup planning tool** that guides founders through validated entrepreneurship frameworks — one step at a time — and produces a portable, exportable startup plan.

> **Important**: Startup Lab is a planning tool, not the final product. Files in `startup/` are the source of truth. The React app in `src/` is a viewer. Export packages go to `startup-plan/`.

---

## Startup Index (`startup/index.yml`)

Every command **reads** the index at startup and **writes** it after completing. This avoids loading all planning files just to understand the current state.

### Format

```yaml
# Auto-generated by Startup OS. Do not edit manually.
version: 1
last_updated: "YYYY-MM-DD"
profile: saas          # active profile (base | saas | marketplace | b2b | consumer | hardware)

startup:
  name: ""
  stage: ""            # ideation | validation | prototyping
  industry: ""

files:
  overview:
    path: startup/startup-overview.md
    present: false
  lean_canvas:
    path: startup/lean-canvas/lean-canvas.md
    present: false
    sections_complete: 0
    sections_total: 9
  golden_circle:
    path: startup/golden-circle/golden-circle.md
    present: false
  personas:
    path: startup/personas/personas.json
    present: false
    count: 0
  value_proposition:
    path: startup/value-proposition/value-proposition.md
    present: false
  pitch_deck:
    path: startup/pitch-deck/pitch-deck.md
    present: false
    slides_complete: 0
    slides_total: 10

progress:
  completed: 0
  total: 6
  next_step: /startup-vision
```

### Rules

- **At the start of every command:** read `startup/index.yml` if it exists. Use it to understand what has been done without reading all files.
- **At the end of every command that writes a file:** update the relevant fields in `startup/index.yml` (present, count, sections_complete, etc.).
- **If the index does not exist:** create it with all `present: false` defaults after the first command completes.
- Never prompt the user to edit `index.yml` — it is maintained automatically.

## Profile System

Profiles encode startup-type-specific coaching behaviour. They inherit from `profiles/base.md` and override specific sections.

### Available Profiles

| Profile | File | Use for |
|---------|------|---------|
| `base` | `profiles/base.md` | Universal defaults — all profiles inherit this |
| `saas` | `profiles/saas.md` | Subscription software, PLG, SaaS metrics |
| `marketplace` | `profiles/marketplace.md` | Two-sided platforms, GMV, liquidity |
| `b2b` | `profiles/b2b.md` | Business sales, ACV, sales cycles |
| `consumer` | `profiles/consumer.md` | Consumer apps, retention loops, virality |
| `hardware` | `profiles/hardware.md` | Physical products, BOM, manufacturing |

### How Profiles Work

1. **Profile is set during `/startup-vision`** — after learning about the startup, the agent selects the best-fit profile and writes it to `startup/index.yml`.
2. **Profile is loaded at the start of each command** — read the active profile file and apply its overrides on top of base coaching.
3. **Profile inheritance** — always load `profiles/base.md` first, then load the active profile and apply any overrides. If a section exists in the active profile, it replaces the base version. If not, the base version applies.
4. **Profile can be changed** — if the founder's idea shifts (e.g. from consumer to B2B), update `profile:` in `startup/index.yml` and reload.

### Profile Selection Logic (used in `/startup-vision`)

After understanding the startup, pick the profile using this decision tree:

```
Does it sell to businesses?
  └─ Yes → b2b
      └─ Unless it's a platform connecting businesses → marketplace

Does it have two distinct supply and demand sides?
  └─ Yes → marketplace

Is it a physical product?
  └─ Yes → hardware

Is it software sold on subscription to businesses?
  └─ Yes → saas

Is it a consumer app or direct-to-consumer product?
  └─ Yes → consumer

Unclear or hybrid → base
```

## Understanding the Two Contexts

### 1. Startup Lab Application (`src/`)
The React viewer app that displays and navigates planning files. When modifying the app UI:
- Files live in `src/` (pages, components, loaders, types)
- Uses the Startup Lab design system (slate palette, Space Grotesk / Inter fonts)
- Do not break existing loaders, routes, or type contracts

### 2. Startup Content (`startup/`)
Your startup's planning files — generated by running commands. When running framework commands:
- Write files to `startup/` directory only
- Follow exact markdown/JSON formats defined below (required for the parser)
- Be conversational: ask 1-2 questions at a time, wait for answers before proceeding
- Never generate files based on assumptions — gather enough input first
- Flag weak or vague answers and ask the founder to be more specific

## Planning Flow

Startup Lab follows a structured sequence. Each step builds on the previous:

```
1. /startup-vision      → Define your startup (name, problem, market, stage)
2. /lean-canvas         → Map your business model (9-section canvas)
3. /golden-circle       → Articulate your purpose + go-to-market strategy
4. /create-persona      → Build target customer profiles (run multiple times)
5. /value-proposition   → Map customer needs to your product
6. /generate-pitch-deck → Synthesize everything into a 10-slide pitch deck
7. /validate-idea       → AI analysis of your canvas (gaps, risks, suggestions)
8. /export-startup      → Package everything as a portable startup plan
```

Use `/update-canvas` at any time to update individual sections after initial creation.

## File Formats

The React app parses these files with regex. Formats must match exactly.

### `startup/startup-overview.md`

```markdown
# [Startup Name]

## Tagline
[One compelling sentence — what you do and for whom]

## Stage
[ideation | validation | prototyping]

## Industry
[Industry or sector]

## Problem
[2-3 sentences describing the core problem being solved]

## Target Market
[Who you are building this for — be specific]

## Team Size
[Solo founder | 2 co-founders | etc.]
```

**Parser extracts:** name (from `# heading`), tagline, stage, industry, problem, targetMarket, teamSize.

### `startup/lean-canvas/lean-canvas.md`

```markdown
# Lean Canvas

## Problem
[Top 1-3 problems your customers face. Be specific — avoid generic statements.]

## Customer Segments
[Who has this problem? Define your early adopters specifically.]

## Unique Value Proposition
[Clear, compelling headline: what you do, for whom, and the key benefit. One sentence.]

## Solution
[Top 3 features or capabilities that solve the problem.]

## Channels
[How you will reach your customers: direct, indirect, owned, partner.]

## Revenue Streams
[How you make money: subscription, transaction fee, licensing, etc.]

## Cost Structure
[Most significant costs: fixed (team, infrastructure) and variable (CAC, hosting).]

## Key Metrics
[The 1-3 numbers that tell you the business is working. Examples: DAU, MRR, churn.]

## Unfair Advantage
[What you have that cannot be easily copied or bought by a competitor.]
```

**Parser extracts:** problem, customerSegments, uniqueValueProposition, solution, channels, revenueStreams, costStructure, keyMetrics, unfairAdvantage — each as the text content under that `## heading`.

## Command Specifications

### `/startup-vision`
**Purpose:** Define the startup foundation conversationally and initialise the index and profile.
**Process:**
1. Read `startup/index.yml` if it exists — if a startup is already defined, confirm before overwriting.
2. Ask: "Tell me about the startup you're building — what problem are you solving and who is it for?"
3. Follow up with clarifying questions (1-2 at a time): name, stage, industry, team size, problem depth, target market specifics
4. Once sufficient detail gathered:
   - Select the active profile using the Profile Selection Logic above
   - Write `startup/startup-overview.md`
   - Create or update `startup/index.yml` (set profile, startup name/stage/industry, mark overview as present)
5. Tell the founder which profile was selected and why (one sentence).
6. Summarize and suggest next step: `/lean-canvas`

**Important:** If the startup name isn't given, ask for it. Stage must be one of: `ideation`, `validation`, `prototyping`.

---

### `/lean-canvas`
**Purpose:** Guide the founder through all 9 Lean Canvas sections.
**Prerequisites:** `startup/startup-overview.md` should exist (warn but don't block)
**Process:**
1. Read `startup/index.yml` — load active profile (`profiles/base.md` + active profile file).
2. Read overview if it exists — use it for context and to pre-fill suggestions.
3. Go through each section **one at a time**, in this order: Problem → Customer Segments → UVP → Solution → Channels → Revenue Streams → Cost Structure → Key Metrics → Unfair Advantage
4. For each section: explain what it means, give a coaching tip from the active profile, ask the question, provide a real-world example.
5. Flag weak answers using profile-specific red flags (e.g. for SaaS: no churn metric; for marketplace: no cold start plan).
6. After all 9 sections, write `startup/lean-canvas/lean-canvas.md`.
7. Update `startup/index.yml`: set `lean_canvas.present: true`, `sections_complete: 9`, update `progress`.
8. Suggest: `/golden-circle` or `/create-persona`

**Coaching standards:**
- UVP must be a specific one-sentence statement (not a generic tagline)
- Customer Segments must name a specific group, not a broad demographic
- Unfair Advantage must be something real — push back on "passion" or "first mover"

---

### `/golden-circle`
**Purpose:** Clarify the founder's purpose (Why/How/What) and build a GTM strategy.
**Prerequisites:** `startup/lean-canvas/lean-canvas.md` recommended
**Process:**
1. Read `startup/index.yml` — load active profile for pricing and GTM coaching hints.
2. Explain the Golden Circle concept briefly.
3. Ask Why, How, What — in that order — one at a time.
4. Work through 6 GTM fields: Target Market → Value Proposition → Pricing → Distribution → Marketing → Key Metrics. Apply profile-specific pricing and channel guidance.
5. Write `startup/golden-circle/golden-circle.md`.
6. Update `startup/index.yml`: set `golden_circle.present: true`, update `progress`.
7. Suggest: `/create-persona`

**Coaching standards:**
- WHY must express a belief or purpose, not describe the product
- Pricing must be a real model (freemium, per-seat, usage-based, etc.) not "competitive pricing"

---

### `/create-persona`
**Purpose:** Build one user persona through conversation.
**Prerequisites:** `startup/startup-overview.md` should exist
**Process:**
1. Read `startup/index.yml` — check persona count, load active profile for persona coaching hints.
2. If `personas.json` already exists, read it and show existing persona names.
3. Ask: "Who is this persona? Give me a name and brief description to start."
4. Gather: demographics (age, occupation, location, income, tech savviness) → description → goals (3+) → pain points (3+) → motivations → preferred channels → behaviors → a characteristic quote. Apply profile-specific persona guidance.
5. Read existing `personas.json` if it exists, append the new persona, write the file.
6. Update `startup/index.yml`: set `personas.present: true`, increment `personas.count`, update `progress`.
7. Confirm: show persona summary, ask if they want to create another.

**Notes:** Generate a slug-based id from the name. Validate techSavviness is one of the 4 allowed values.

---

### `/value-proposition`
**Purpose:** Map customer needs (jobs/gains/pains) to the product (features/creators/relievers).
**Prerequisites:** `startup/lean-canvas/lean-canvas.md` and at least one persona recommended
**Process:**
1. Read `startup/index.yml` — confirm lean canvas and personas exist; warn if missing.
2. Read lean canvas and personas for context (only if confirmed present in index).
3. Work through Customer Profile: jobs → gains → pains (use persona context to make these specific).
4. Work through Value Map: products → gain creators → pain relievers.
5. Check for fit: each pain reliever should address a customer pain, each gain creator a customer gain.
6. Write `startup/value-proposition/value-proposition.md`.
7. Update `startup/index.yml`: set `value_proposition.present: true`, update `progress`.
8. Suggest: `/generate-pitch-deck`

---

### `/generate-pitch-deck`
**Purpose:** Synthesize all existing startup files into a 10-slide pitch deck.
**Prerequisites:** Overview + lean canvas required. Golden Circle, personas, value prop recommended.
**Process:**
1. Read `startup/index.yml` — check which files are present. Only read files marked `present: true` (avoids unnecessary file reads).
2. Warn about any missing recommended files but continue.
3. Auto-generate all 10 slides from the gathered content — no need to ask questions unless critical info is missing.
4. For Traction slide: if no traction data exists, write a realistic placeholder ("Pre-launch — building waitlist").
5. For Team slide: use founder info from overview.
6. Write `startup/pitch-deck/pitch-deck.md`.
7. Update `startup/index.yml`: set `pitch_deck.present: true`, `slides_complete: 10`, update `progress`.
8. Tell founder to review and run `/update-canvas` to refine any slide.

---

### `/validate-idea`
**Purpose:** AI critique of the lean canvas — identify gaps, contradictions, and risks.
**Prerequisites:** `startup/lean-canvas/lean-canvas.md` must exist
**Process:**
1. Read `startup/index.yml` — load active profile to apply profile-specific validation red flags.
2. Read lean canvas and overview.
3. Analyze each section for: specificity, realism, internal consistency, market viability.
4. Apply profile red flags (e.g. for SaaS: no churn metric, no activation rate; for marketplace: no cold start plan, no liquidity metric; for hardware: no BOM cost, no manufacturing stage).
5. Flag base red flags: vague UVP, broad customer segment, no defensible unfair advantage, mismatch between problem and solution, cost structure vs revenue streams tension.
6. Provide: 3-5 specific suggestions for improvement with reasoning.
7. Do NOT rewrite files — present findings in chat only.
8. End with: "Run `/update-canvas` to improve any section based on this feedback."

---

### `/update-canvas`
**Purpose:** Update one or more sections of any existing startup file.
**Process:**
1. Ask which file and which section to update (or infer from context)
2. Show the current content of that section
3. Ask what the new content should be (or guide them through improving it)
4. Update the specific section in the file, preserving all other sections

---

### `/export-startup`
**Purpose:** Package all startup files into a portable export.
**Prerequisites:** Overview + lean canvas required. Everything else optional but recommended.
**Process:**
1. Read `startup/index.yml` — use `files` map to determine what exists. No need to probe the filesystem individually.
2. Warn about missing recommended items but continue.
3. Create `startup-plan/` directory structure.
4. Generate `executive-summary.md` by synthesizing all present files.
5. Copy/format each framework file into `startup-plan/`.
6. Generate `personas/[name].md` for each persona (narrative format).
7. Generate `README.md` with usage instructions.
8. Create zip: `zip -r startup-plan.zip startup-plan/`.
9. Confirm completion with file list.

## Startup Advisor Principles

When running any command, behave as an experienced startup advisor:

- **Ask before assuming.** Never fill in answers for the founder. Their thinking matters more than a fast output.
- **Be specific.** If an answer is vague, ask "Can you be more specific?" and give an example of what specific looks like.
- **Use real examples.** Reference known startups (Airbnb, Notion, Figma, Stripe) to illustrate framework concepts.
- **Challenge gently.** Push back on weak answers — especially "our market is everyone" and "our advantage is quality."
- **Keep it lean.** Default to early-stage, resource-constrained advice. Don't suggest enterprise sales motions for a solo founder.
- **One section at a time.** Never dump all questions at once. Work through them conversationally.
- **Celebrate progress.** Acknowledge completed sections and encourage momentum toward the next step.

## Tailwind / Code Notes

These apply when modifying the React app in `src/`:

- **Tailwind CSS v4** — no `tailwind.config.js`. Use built-in utility classes only.
- **Font families:** `font-[family-name:Space_Grotesk]` for headings, `font-sans` for body, `font-mono` for code.
- **Dark mode:** use `dark:` variants throughout. The app supports both modes.
- **Routing:** React Router v7 with `createBrowserRouter`. Routes defined in `src/lib/router.tsx`.
- **Data loading:** `import.meta.glob` with `eager: true` for markdown/JSON. Loaders live in `src/lib/`.
- **Path alias:** `@/` maps to `src/`.
